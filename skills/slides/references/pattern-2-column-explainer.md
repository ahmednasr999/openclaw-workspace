# Pattern: Two-Column Explainer

## Best for
- explaining a concept with supporting evidence
- showing statement + proof
- pairing narrative with visual or detail block

## Use when
- the slide has one main idea and one supporting region
- the two sides do different jobs

## Don’t use when
- both sides are truly equal and should be treated as a comparison instead
- the slide would be stronger as a single-message hero or metric page

## Required content inputs
- primary statement or explanatory block
- secondary support block

## Optional content inputs
- proof points
- mini diagram
- quote, evidence, or callout

## Layout logic
- one side must dominate
- the second column exists to support, not compete
- avoid perfectly balanced columns by default

## Hierarchy rules
- dominant message first
- support block second
- tertiary notes or metadata last

## Focal-point rule
The focal point is the primary explanatory block. If both columns shout equally, the slide loses hierarchy.

## Visual treatment guidance
- asymmetry is preferred
- support side can be a module, image field, or evidence panel
- use whitespace to keep the relationship clear

## Common anti-patterns
- two equal columns with no dominance
- too much text in both columns
- support block becoming a cluttered card stack

## Example structures
- left thesis, right evidence module
- left narrative, right visual block
- left recommendation, right implementation note

## QA checklist
- which side dominates?
- does the support side stay subordinate?
- would the slide improve if one block were simplified?
