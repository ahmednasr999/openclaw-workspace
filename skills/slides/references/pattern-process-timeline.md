# Pattern: Process Timeline

## Best for
- roadmap slides
- phased plans
- implementation sequence
- process explanation with clear flow

## Use when
- the audience needs to understand sequence or phase structure
- timing, ordering, or step progression is the point of the slide

## Don’t use when
- the content is really a comparison, not a sequence
- there are too many steps for one slide to carry clearly

## Required content inputs
- sequence of steps, phases, or milestones

## Optional content inputs
- owners
- timing labels
- one key dependency or blocker note

## Layout logic
- one strong directional flow
- steps should read in order without confusion
- use grouping when the process has phases
- avoid compressing too many nodes into one horizontal strip

## Hierarchy rules
- overall process title first
- phase or step labels second
- supporting details third

## Focal-point rule
The focal point is the process spine or the current decisive phase, not a cluster of equal micro-boxes.

## Visual treatment guidance
- use spacing to show sequence clearly
- allow one emphasized phase or milestone
- connectors should clarify, not decorate

## Common anti-patterns
- too many tiny steps in one row
- no obvious start or finish
- every milestone carrying equal visual weight
- process slide that feels like a Gantt dashboard without enough hierarchy

## Example structures
- 4-step horizontal phase sequence
- 3-phase roadmap with one highlighted current phase
- vertical process flow with grouped sections

## QA checklist
- is the order obvious immediately?
- can at least one phase be emphasized more clearly?
- should this sequence be split across two slides?
