## Model Routing Fix — Complete (19:15 Cairo)

**Problem:** Telegram DMs kept routing to MiniMax-M2.7 instead of Anthropic/claude-opus-4-6, despite `channels.modelByChannel.telegram = {"*": "anthropic/claude-opus-4-6"}` in openclaw.json.

**Root causes (multiple layers):**

1. **Session model override persistence:** Gateway writes fallback model (MiniMax) back to sessions.json as `modelOverride`/`providerOverride`. On subsequent messages, `createModelSelectionState()` reads these stored overrides and overwrites the channel override. Channel override applied → then stored model override overwrote it.

2. **Session authProfileOverride:** Session had `authProfileOverride: "minimax-portal:default"` which forced MiniMax OAuth even after model was changed.

3. **Gateway code bug:** `createModelSelectionState()` (model-selection-CDZG0zcK.js) has `skipStoredOverride` flag that only triggers for heartbeat overrides (`hasResolvedHeartbeatModelOverride`), not channel overrides.

**Fix applied (TWO files patched in OpenClaw dist):**
- `/usr/lib/node_modules/openclaw/dist/reply-CxEVitwF.js` — After channel override block (line ~2966): added `hasResolvedChannelModelOverride` flag + preserved `channelOverrideProvider` + `channelOverrideModel`. Passed all three to `createModelSelectionState()`.
- `/usr/lib/node_modules/openclaw/dist/model-selection-CDZG0zcK.js` — Line ~230: Changed `const skipStoredOverride = params.hasResolvedHeartbeatModelOverride === true` → `const skipStoredOverride = params.hasResolvedHeartbeatModelOverride === true || params.hasResolvedChannelModelOverride === true`

**Session cleanup:** Cleared `modelOverride`, `providerOverride`, `authProfileOverride`, `authProfileOverrideSource`, `authProfileOverrideCompactionCount` from session `agent:main:telegram:direct:866838380` in sessions.json.

**Result confirmed live:** Session now reports `Model: anthropic/claude-opus-4-6 · 🔑 token (anthropic:manual)` — Opus 4.6 via Anthropic API key, working.
