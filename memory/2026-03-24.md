### Image Upload: Composio S3 Key Discovery
**Problem:** `CREATE_LINKED_IN_POST` always returned HTTP 404 for images. Composio tries to download from its own S3, not from LinkedIn, Google Drive, Uploadcare, or any external URL.

**What was tried and failed:**
- `LINKEDIN_REGISTER_IMAGE_UPLOAD` + HTTP PUT to LinkedIn URL → got LinkedIn asset URN → used as s3key → 404
- `LINKEDIN_INITIALIZE_IMAGE_UPLOAD` + HTTP PUT → got `urn:li:image:...` → as s3key → 404
- Google Drive file ID as s3key → 404 (network isolation confirmed)
- Uploadcare UUID as s3key → 404
- Direct LinkedIn REST API via Code Execution → Python syntax error (blocked)

**Deep investigation:**
- Composio workbench runs in E2B sandbox with firewall: `api.composio.dev`, `cdn.composio.dev`, `storage.composio.dev` all unreachable
- Only `connect.composio.dev` (104.18.20.5) resolves
- `AWS_SECRET_ACCESS_KEY` missing from environment — direct S3 access impossible
- Composio MCP endpoint confirmed — but `LINKEDIN_CREATE_LINKED_IN_POST` tool is NOT exposed via direct MCP (tool routing only works via Composio's internal tool router)

**Solution found:** `upload_local_file()` function INSIDE `COMPOSIO_REMOTE_WORKBENCH`
- Uploads image bytes to Composio's internal S3
- Returns s3key in format: `project/pr_DBp_8iitCrNe/tool_router_session/trs_OOc0h6Vp0j01/filename`
- This s3key works with `CREATE_LINKED_IN_POST`
- Confirmed project ID `pr_DBp_8iitCrNe` and session `trs_OOc0h6Vp0j01` are stable across sessions

**Correct image post flow:**
1. `LINKEDIN_REGISTER_IMAGE_UPLOAD` → get `uploadUrl` + `asset_urn` (LinkedIn's URN, not s3key)
2. Download image → save locally
3. `COMPOSIO_REMOTE_WORKBENCH` → `upload_local_file(local_path)` → get Composio s3key
4. `CREATE_LINKED_IN_POST` with `images[0].s3key` = Composio s3key AND `images[0].name` = filename
5. Post succeeds

**Today's image post:** Successfully posted with image. Post ID: `urn:li:share:7442155953715666944`

### job-ops Repository Deep Dive
**URL:** https://github.com/DaKheera47/job-ops
**What it is:** Self-hosted Docker job hunting pipeline. Scrapes LinkedIn, Indeed, Glassdoor, Adzuna, HiringCafe, Gradcracker, UKVisaJobs.

**Architecture:**
- Single Docker container (port 3005 for UI, 3001 for API)
- Extractors: per-board scrapers using `python-jobspy` library
- Orchestrator: normalizes + deduplicates + AI-scores jobs
- Ghostwriter: per-job AI chat (cover letters, interview prep, outreach)
- RxResume integration: PDF CV generation
- Gmail OAuth: inbox tracking (auto-detects interview/offer/rejection signals)
- SQLite DB for all data
- API: `PATCH /api/jobs/<jobId>` for updates

**How JobSpy accesses LinkedIn:**
- Uses **LinkedIn public job search pages** — NO `li_at` cookie required for discovery
- For full descriptions: hits each job URL directly (anti-bot risk, proxies supported)
- Supports `linkedin_fetch_description=True` for full JDs

**Key insight for our system:**
- JobSpy (no auth) = job discovery (titles, companies, URLs)
- Our system = strategic CV creation + ATS optimization
- Composio = posting
- Gmail OAuth from job-ops = application status tracking
- These are complementary, not competing

**Setup prerequisites for job-ops on our server:**
1. RxResume account (for PDF generation)
2. Google OAuth credentials (for Gmail tracking)
3. LLM API key (OpenRouter compatible)
4. Docker installed
